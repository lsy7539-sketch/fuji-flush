# Fuji Flush

3~8명이 플레이하는 핸드 셰딩 카드 게임. 손패를 가장 먼저 모두 없애면 승리.

배포 주소: https://fuji-flush.onrender.com (Render.com 무료 플랜, GitHub `lsy7539-sketch/fuji-flush`
연결, main 브랜치에 푸시하면 자동 재배포됨)

## 현재 상태

- 게임 엔진 + 단위 테스트 완료. 디자인은 3개 시안 중 "카지노 펠트" 테마로 확정.
- 접속하려면 입장 코드가 필요함 (관리자가 발급). 관리자 화면은 `#admin` 해시로 접근.
- 두 가지 플레이 모드 구현 완료:
  - **혼자하기 (AI 상대)**: `npm run dev` 로 Vite 개발 서버를 띄우고 브라우저에서 바로 플레이.
  - **온라인 멀티플레이**: `npm run build` 후 `npm run server` 로 Node 서버를 띄우면 같은 주소로
    여러 사람이 접속 가능.
- 게임 중 화면 우상단에 뒤로가기(←)/일시정지(⏸)/나가기(✕) 컨트롤 있음.
- `npm test` 로 테스트, `npm run typecheck` 로 타입 체크.

## 구조

- `src/engine/` — 게임 규칙 (수정 없음, 모든 모드가 그대로 재사용)
  - `types.ts`, `deck.ts`, `gameEngine.ts` (`createGame`/`playCard`/`getActiveGroups`)
  - `playerView.ts` — `toPlayerView(state, viewerId)`: 손패를 본인 것만 보여주고 나머지는
    장수만 노출하는 뷰. 로컬 모드(봇 손패 숨기기)와 서버(실제 보안 경계) 둘 다 이 함수 하나로 처리.
- `src/ai/botPlayer.ts` — `chooseBotMove(state, playerId)`. `playCard`를 순수 dry-run
  시뮬레이터로 사용해 각 후보 카드를 시험해보고, 안전한 카드 중 가장 낮은 걸 내거나(안전한 게
  없으면 결과 총합이 가장 높은 걸) 선택. 규칙 판정 로직은 전혀 재구현하지 않음.
- `src/shared/protocol.ts` — 클라이언트/서버 WebSocket 메시지 타입 (`ClientMessage`/`ServerMessage`).
  로그인/관리자 기능은 WebSocket이 아니라 별도의 REST 엔드포인트(`/api/...`)로 분리되어 있음 —
  방 입장 전에 한 번 확인하고 끝나는 요청이라 게임 프로토콜과 섞을 이유가 없었음.
- `src/server/` — 온라인 멀티플레이 + 인증 서버 (Express + `ws`, `tsx`로 실행)
  - `server.ts` — `dist/` 정적 파일 서빙 + `/ws` WebSocket 업그레이드 + `/api/login`,
    `/api/admin/*` REST 라우트. 전부 한 포트로 통합.
  - `rooms.ts` — 방/로비 상태를 메모리에 `Map`으로 보관. 서버가 유일하게 `createGame`/`playCard`를
    호출하는 주체 (클라이언트는 절대 게임 상태를 직접 계산하지 않음). 모든 브로드캐스트는 플레이어별로
    `toPlayerView`를 따로 계산해서 보냄 — 절대 하나의 공유 상태를 그대로 뿌리지 않음.
  - `roomCode.ts` — 5자리 방 코드 생성 (혼동되는 0/O, 1/I 제외). 방 만들 때 직접 코드/이름을
    지정할 수도 있음 (비우면 이 생성기가 자동으로 채움).
  - `accessCodes.ts` — 로그인용 입장 코드 저장소 (방 코드와는 별개 개념, 관리자가 직접 문자열을
    정해서 등록/삭제). `db.ts`를 통해 영속 저장됨.
  - `db.ts` — Turso(프로덕션)/로컬 SQLite 파일(개발) 클라이언트 + 테이블 초기화. `TURSO_DATABASE_URL`
    환경변수가 없으면 자동으로 `./data/fuji-flush.db` 로컬 파일을 씀 — Turso 계정 없이도
    `npm run dev`/`npm run server`가 바로 됨.
- `src/client/` — 프론트엔드
  - `main.ts` — 진입점. `#admin` 해시면 관리자 화면, 로그인 안 됐으면 입장 코드 화면, 그 다음
    모드 선택 화면(`혼자하기`/`온라인 멀티플레이`) 순으로 라우팅.
  - `loginGate.ts` — 입장 코드 입력 화면. 성공하면 `sessionStorage`에 인증 플래그 저장(탭 닫으면
    다시 코드 필요).
  - `adminMode.ts` — 관리자 비밀번호 로그인 + 입장 코드 목록/생성/삭제 화면.
  - `render.ts` — `PlayerFacingState`를 받아 보드를 그리는 순수 렌더 함수 (모든 모드가 공유).
    뒤로가기/일시정지/나가기 버튼도 여기서 렌더링하고, 콜백은 각 모드가 채워넣음.
  - `localMode.ts` — 로컬 `GameState` 보유, 봇 턴은 500ms 딜레이로 한 수씩 진행. 일시정지 시
    봇 타이머도 멈춤.
  - `networkMode.ts` — 방 만들기/참가 로비 화면 + WebSocket 클라이언트. 카드 클릭 시 서버에
    `playCard` 메시지만 보내고, 다음 `stateUpdate`가 올 때까지 버튼을 비활성화 (낙관적 UI 없음).

## 핵심 설계 결정

- **그룹(Joining Forces)은 상태로 저장하지 않고 매번 계산한다.** `activeCards`를 raw `value`로
  묶으면 그룹이 결정되므로 `groupId`는 `group-<value>`로 매번 새로 유도. 과거 이력을 들고 다닐
  필요가 없어 동기화 버그 위험이 없음.
- **Player에 `activeCards` 필드를 두지 않았다.** `GameState.activeCards`가 이미 `playerId`로
  추적하므로 중복 저장하면 두 값이 어긋날 위험만 생김.
- **`playCard(state, playerId, cardId?)`에서 `cardId`는 optional.** 손패 0장인 채로 자기 차례가
  됐는데 이전에 낸 카드가 Pushed Through되며 바로 승리하는 경우(규칙 14)가 있어서, 낼 카드가
  없어도 턴이 정상 종료돼야 함.
- **`getActiveGroups`는 `GameState`가 아니라 `ActiveCard[]`를 받는다.** 그래야 손패 정보가 없는
  `PlayerFacingState.activeCards`에도 같은 함수를 그대로 재사용할 수 있음 (그룹 계산 로직 중복 방지).
- **손패 숨김은 `toPlayerView` 하나로 통일.** 로컬 모드의 "봇 손패 숨기기"는 단순 연출이지만,
  서버의 "다른 사람 손패 숨기기"는 실제 보안 경계임 — 같은 함수를 재사용해서 두 맥락에서 동일하게
  검증됨.
- **다른 플레이어는 화면 위쪽에 이름+장수만 표시, 내 손패만 크게 배치.** 상대 손패를 카드로
  펼쳐 보여줄 이유가 없어서(어차피 안 보이는 카드) `render.ts`가 `viewer`와 `opponents`를
  분리해서 서로 다른 컴포넌트로 그림.
- **뒤로가기/일시정지/나가기의 의미가 모드마다 다르다.** 혼자하기: 뒤로가기=인원수 재설정 화면
  (확인 없음), 나가기=모드선택 화면(확인 후). 멀티플레이: 뒤로가기=현재 방만 나가고 로비 화면으로
  (확인 후), 나가기=모드선택까지(확인 후) — 실제 사람들이 걸려있는 세션이라 둘 다 확인을 거침.
  일시정지는 혼자하기에서는 봇 타이머까지 실제로 멈추지만, 멀티플레이에서는 내 화면만 어둡게/입력
  차단할 뿐 다른 사람의 게임은 멈추지 않음 (애초에 공유 세션을 한 명이 멈출 방법이 없음).
- **입장 코드(access code)와 방 코드(room code)는 별개 개념이다.** 입장 코드는 "이 앱에 아예
  들어올 수 있는지"를 가르는 관문(관리자만 발급), 방 코드는 "이미 들어온 사람들끼리 같은 게임에
  모이는" 용도. 그래서 저장소도 `accessCodes.ts`/`rooms.ts`로 분리했고, 길이도 다르게 뒀음
  (8자리 vs 5자리).
- **로그인/관리자 인증은 세션 쿠키가 아니라 매 요청 헤더/바디로 비밀번호를 보내는 단순한 방식.**
  `sessionStorage`에 인증 여부(사용자)와 관리자 비밀번호 원문(관리자)을 저장해두고 매 관리자
  API 요청에 `x-admin-password` 헤더로 실어 보냄. 개인 프로젝트 규모에 맞춘 단순화이고,
  실제 서비스 수준의 보안(해시된 세션 토큰, HTTPS 전제하의 안전성 등)은 아님 — 이미 HTTPS로
  배포되어 있어 전송 구간 도청 위험은 낮지만, 알아두어야 할 트레이드오프.
- **입장 코드와 게임 전적(matches/match_players)만 DB에 영속 저장, 방/로비 상태는 여전히
  메모리에만 있다.** 진행 중인 방은 서버가 재시작되면 어차피 모든 소켓 연결이 끊겨서 플레이어들이
  다시 들어와야 하므로(재접속 미구현) 방 상태 자체를 영속화할 이유가 없음. 반면 입장 코드(계속
  써야 함)와 끝난 게임의 전적(나중에 랭킹/통계용)은 재시작에도 남아야 해서 DB로 뺐음.
- **게임이 끝나는 순간(`gameStatus === "FINISHED"`) `rooms.ts`가 비동기로(await 안 하고) 전적을
  DB에 기록한다.** 플레이어들에게 결과를 보여주는 브로드캐스트가 DB 쓰기를 기다릴 이유가 없어서
  fire-and-forget으로 처리 (실패하면 콘솔에 로그만 남김). 랭킹/통계 화면 자체는 아직 없고, 데이터만
  쌓아두는 단계임.
- **재접속(reconnect)은 의도적으로 구현하지 않았다.** 게임 중 탭이 새로고침되거나 연결이 끊기면
  그 자리는 복구할 방법이 없음.

## 다음 할 일

- **⚠️ Render.com 환경변수에 `ADMIN_PASSWORD` 반드시 설정할 것.** 코드 기본값은
  `"changeme-admin"`이라 실배포 상태로 그냥 두면 위험함. Render 대시보드 → 서비스 → Environment
  탭에서 설정.
- **⚠️ Render.com 환경변수에 `TURSO_DATABASE_URL`/`TURSO_AUTH_TOKEN` 설정할 것.** 안 하면
  로컬 파일(`./data/fuji-flush.db`)로 자동 대체되는데, Render는 배포/재시작마다 파일시스템이
  초기화되므로 실질적으로 영속되지 않음 — 반드시 실제 Turso DB를 붙여야 입장 코드/전적이 안 사라짐.
- **재접속 지원.** 브라우저 새로고침/연결 끊김 시 같은 자리로 복귀할 수 있게 하려면 플레이어별
  재접속 토큰을 발급해 `localStorage`에 저장하고, 재연결 시 그 토큰으로 같은 좌석에 매핑해야 함.
- **전적/랭킹/통계 화면.** `matches`/`match_players` 테이블에 데이터는 쌓이기 시작했지만, 이를
  보여주는 화면(승률, 랭킹 등)은 아직 없음 — 나중에 만들 것. 참고: 플레이어 식별은 닉네임 문자열
  기준이라, 같은 닉네임을 쓰는 다른 사람은 구분되지 않음 (실제 계정 시스템이 없어서 생기는 한계).
- **로컬 개발 시 멀티플레이 테스트 흐름**: `npm run build` → `npm run server` → 브라우저 탭 여러
  개(또는 같은 와이파이의 다른 기기에서 내 PC의 LAN IP로 접속)로 확인. 이 경로는 핫 리로드가 없어서
  코드 바꿀 때마다 다시 build해야 함.
